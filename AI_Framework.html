Igris

AI-Powered Test Automation Framework

What Is Igris?

The idea for this framework came from a conversation with a friend: “What if we could generate automated scripts from simple words?” From that spark, I built Igris, an AI‑powered testing framework that connects with OpenAI to generate a JSON plan that functions as test cases, using SauceDemo — a standard e-commerce test app — as the demo site. A small team has already started using it and shared feedback, which helped me refine the AI, add new features, and integrate CI/CD through GitHub Actions and workflow automation.

Stack: Python · pytest · Playwright · OpenAI API · Streamlit · Allure · GitHub Actions

See It In Action

Instruction used for this demo:

"Create a test where the user logs in with valid credentials, then adds two products, navigates to the cart, completes the checkout flow, and validates the order was completed successfully."

Step 1 — Plain text input via Streamlit

Run "streamlit run app.py" in cmd, then type the title and scenario to generate the test plan. No code, no selectors, no test structure required.

Step 2 — AI generates the JSON test plan

The agent interprets the instruction and returns a structured plan. Every action maps to something the translator knows how to execute.

Step 3 — Test plan execution

Once the plan is ready, Igris runs it against the browser — or both Chromium and Firefox in parallel, depending on the configuration.

Step 4 — Allure report: suite overview

All runs visible in the Allure dashboard, organized by suite and browser.

Step 4 — Allure report: products verified

Per-step screenshots embedded in the report. Both products from the instruction confirmed as added to the cart.

Step 4 — Allure report: order confirmed

The final assertion verifies the confirmation screen. Test closes with a validated successful transaction.

Step 5 — GitHub Actions

All pull request and push to main triggers the smoke test. Allure report generated and uploaded as a downloadable artifact automatically.

How It Works

Igris uses selectors and prompts to understand the page structure and generate a JSON plan from a plain text instruction. That plan is then converted into a pytest file and executed via Playwright. Everything from browser selection and AI model to screenshots and Allure reports is controlled through a config file — no code changes needed between runs. It's split in two independent layers: the AI generation side calls OpenAI and produces the test file, while the execution side — pytest, Playwright, and CI/CD — runs completely on its own, no API key needed.

For CI/CD, I set up a smoke test that runs whenever code is pushed or a pull request is made. There's a script that runs through the entire checkout flow automatically — so every time someone commits, the test validates the whole flow end-to-end.

Note: depending on the model, instructions may need to be more or less detailed — a more capable model handles vague inputs better, a lighter one benefits from more specific wording.

The Design Decision That Makes It Portable

The most deliberate choice in Igris is what the AI doesn't produce. Instead of generating Python code or Playwright scripts directly, the AI outputs a language-agnostic JSON plan. A separate translator layer reads that plan and executes it.

This is the Interpreter Pattern — and it has a specific purpose: the framework was designed knowing it could be adopted by different teams, in different languages, on different projects.

Igris Agent
Plain text instruction → OpenAI
↓
JSON Plan
Works with any language
↓
AI_plan_translator.py
Current: Python · Playwright
Any future translator
TypeScript · Java · Selenium…

If the project migrates to a different language or framework, only the translator needs to be rewritten — Igris and the JSON plans remain unchanged.

Target What changes What stays the same
TypeScript + Playwright Translator rewritten in TS AI agent, JSON plan, selectors
Java + Selenium Translator + selector resolution AI agent, JSON plan structure
Cypress Translator maps to cy.* calls AI agent, JSON plan
Mobile (Appium) Translator + mobile selectors AI agent, JSON plan structure

The AI layer stays unchanged. The JSON plan stays unchanged. Only the translator is swapped. A team already on Java doesn't need to adopt Python — they write their own translator and plug it in.

What Makes It Reliable

A few specific decisions were made to keep the AI output consistent and the test execution trustworthy:

  • Screen-aware AI — the agent's prompt includes a map of the application's screens in order. The AI knows which selectors belong to which screen and never mixes them up across steps.
  • Constrained actions — the AI can only produce actions the translator knows how to run. No hallucinated methods, no unsupported steps.
  • Assert reasoning — a chain-of-thought rule in the prompt forces the AI to analyze the element type before choosing an assertion. This reduces false positives and makes assertions more meaningful.
  • Deterministic output — the agent runs at temperature 0. The same instruction produces the same plan every time.
  • Dual screenshot storage — every step screenshot is saved to disk in a timestamped folder and embedded in the Allure report. Evidence survives even if the report is regenerated.

The current setup works great for demos and small to mid-size projects. For bigger applications with several flows, multi-agent orchestration and RAG would be the natural next step.
